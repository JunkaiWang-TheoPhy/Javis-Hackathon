# 七、安全防护和隐私设计

## 安全360°：四层防护

更强的感知能力，意味着更大的隐私责任。Mira 的目标并不是“知道得越多越好”，而是在真正需要的时候，以最小打扰、最小暴露、最小越界的方式提供帮助。也正因为如此，安全与隐私并不是系统完成之后再补上的附加条件，而是从架构一开始就被纳入设计边界之中。围绕访问入口、运行能力、AI 行为和终端控制，我们建立了四层相互配合的防护体系，让 Mira 的感知、判断和行动始终处在清晰、可控的范围内。

Mira 的能力建立在持续感知、长期记忆和主动行动之上，因此它面对的安全与隐私问题，并不只是“数据会不会泄露”，而是一个更完整的问题：谁可以接近系统、系统可以做多少事、模型在什么边界内行动、用户是否始终保有最终控制权。换句话说，Mira 的安全设计不是在系统之外额外补上的一层外壳，而是决定它能否被真正信任的系统基础，也是整个系统可信边界（trust boundary）的一部分。

我们并不认为，一个真正有用的陪伴系统应该以“尽可能多感知、尽可能多保存、尽可能多执行”为目标。相反，Mira 追求的是另一种更克制的能力：在真正需要的时候理解，在边界允许的时候行动，在不需要的时候保持沉默。围绕这个目标，四层防护分别守住系统的不同风险面（risk surface），也共同回答了一个更重要的问题：Mira 为什么能够主动，但又不会轻易越界。

**L1 云服务器安全（Access Security）**

第一层解决的是“谁可以进入系统”的问题。Mira 的远程访问、管理入口和服务连接都应当处于受控环境中，通过身份验证、访问限制和加密传输来降低暴露面（attack surface），确保只有被授权的人和设备，才能接近 Mira 的核心运行环境。对于一个具备持续感知和主动行动能力的系统来说，先守住入口，才能守住后面的所有边界；如果最外层入口失守，再精细的策略也可能失去意义。

这一层真正要防的，不是单一的技术漏洞，而是“系统在还没有建立信任关系之前，就被过早暴露出去”。Mira 所面对的不是一个孤立的本地程序，而是一整套与眼镜、手机、手表、家居和通知系统相互连接的运行环境。只要任何人、任何设备、任何通道都可以在缺乏识别和约束的情况下接近它，那么后面再细致的权限控制和行为约束，都可能变得被动。因此，L1 的职责不是增加功能，而是减少暴露面；不是让系统变得更开放，而是让系统先学会拒绝不必要的接近。

从设计理念上看，这一层实际上建立了 Mira 与外部世界之间的第一条信任边界（trust boundary）。普通用户的使用入口、维护者的管理入口、系统服务之间的通信路径，本来就不应该混在同一个平面上。越是涉及持续感知和主动行动的系统，越不能把“方便接入”当成第一优先级。Mira 必须首先确认“谁能进来”，然后才谈“进来以后能做什么”。也正因为如此，L1 并不是外围附属，而是整个安全体系的起点，也是系统身份面（identity plane）与连接面（connection plane）的基础保护层。

**L2 OpenClaw 加固（Runtime Hardening）**

第二层解决的是“即使进入系统，也不能无限制行动”的问题。Mira 依托 OpenClaw 运行，但不会默认开放所有底层能力，而是通过运行时约束、命令范围限制、最小权限原则（principle of least privilege）和审计机制，把系统能力收敛在必要范围内。换句话说，Mira 并不是一个可以随意调用底层资源的 agent，而是一个在明确边界内运行、并且具备可追踪性的受限系统。这一层的作用，是避免感知、执行和系统权限被无边界地交给模型或单一服务，让系统的主动性建立在可信的运行底座之上。

这一层之所以必要，是因为 Mira 所连接的不是抽象文本，而是眼镜、手机、手表、消息系统、家居控制和真实设备等执行面。一旦运行时边界过宽，风险就不再只是信息层面的泄露，而会进一步扩展为现实世界中的错误动作与越界干预。因此，Mira 在运行时层面强调 Shell 沙箱封装、权限最小化、安全审计以及能力隔离（capability isolation），并不是为了让系统“什么都不能做”，而是为了让它“只能做被允许的事”。它可以有能力，但这种能力必须始终被边界化。

从更深一层的角度看，L2 体现的是 Mira 对“能力”本身的克制理解。技术系统常常追求“我能做到更多”，但陪伴系统更需要回答的是“我为什么只做这些”。OpenClaw 加固的意义，不是削弱 Mira，而是让 Mira 的强大始终处于可被信任的边界之中。只有当运行时平面（runtime plane）本身就是一条安全边界时，Mira 的主动感知与主动行动，才不会从能力优势滑向能力滥用，也不会从执行风险（execution risk）滑向现实风险。

**L3 AI 行为边界（Behavioral Governance）**

第三层解决的是“即使模型理解了，也不代表它可以直接行动”的问题。Mira 的主动能力不是无条件开放的，而是始终受到行为边界的约束。系统会通过 `rejection.md` 中的禁止行为清单、记忆中的安全规则以及“沉默优先”原则，明确哪些事情不能做，哪些事情需要更谨慎，哪些情况下应当保持沉默。也就是说，Mira 的主动性始终是被约束、被校准的主动性，而不是由模型自由发挥的“热心”。它会尽量先理解，再决定是否需要介入，而不是因为“似乎可以帮忙”就立刻越界行动。

这一层首先守住的，是从“理解”到“行动”之间的那一道门。Mira 的优势，不只是它能够从视觉、生理、环境和历史上下文中识别出疲惫、紧张、孤独、沉默或风险上升的状态，更在于它知道：识别到这些状态，并不意味着自己已经自动获得了行动的正当性。系统仍然需要继续判断，此时更合适的是提醒本人、继续观察、安静调整环境，还是完全不做任何动作。这种“再停一步、再想一步”的机制，正是 Mira 不同于普通自动化系统的地方，也是它决策边界（decision boundary）的核心所在。

因此，Mira 的行为边界并不依赖一次性的提示词，而是通过规则、记忆和策略被持续写入系统本身。什么不能做，什么不该做，什么需要更谨慎地做，不是模型临场发挥出来的，而是被反复强化的边界。真正的照护，不是无时无刻都出现，也不是在每一个信号变化时都迫不及待地表态。很多时候，真正成熟的系统恰恰知道什么时候不要出现，什么时候不应该替用户做决定，什么时候应该把主动性收回去。L3 的意义，就是让 Mira 始终记得：会理解，不等于该行动；能行动，不等于该越界。这也是 Mira 采取默认保守（default conservative）策略的原因。

**L4 终端设备控制（Device Sovereignty & User Control）**

第四层解决的是“最终控制权始终属于谁”的问题。Mira 所连接的是眼镜、手机、手表、电脑和智能家居等真实设备，因此每一项终端能力都必须建立在明确授权的前提下，而不是默认长期开放。设备白名单授权意味着，只有经过确认和信任的设备才能接入 Mira 的感知与执行体系，系统不会对未知设备或未授权能力默认放行。同时，Mira 坚持本地优先（local-first）的数据处理原则，尽可能在本地完成分析，只保留真正必要的结构化结果，而不会无限制地长期保存原始数据；对于视频帧这类敏感信息，也会在完成当前分析后及时销毁。更重要的是，用户始终拥有主动控制权：感知可以暂停，设备连接可以撤销，记忆内容可以清除。Mira 的目标不是接管用户的生活，而是在用户授权的边界内提供更有分寸的陪伴与支持。

这一层的意义在于，它把“系统有能力做什么”重新拉回到“用户允许系统做什么”。陪伴系统天然容易跨进一个危险区域：它越懂你，就越可能被误以为“可以替你决定”。但 Mira 的设计前提恰恰相反。它不是要接管生活，而是要在用户授权、用户可撤回、用户可中止的边界内，提供更柔和、更及时、更有分寸的支持。系统越强，用户就越需要拥有明确、直接、随时可用的控制权（user override）；而一个真正值得信任的系统，也必须把这种控制权视为自己的前提，而不是障碍。

因此，L4 真正体现的是 Mira 的底层立场：感知不是默认权利，而是用户临时授予的能力；数据不是系统资产，而始终属于用户本人；行动不是模型自己决定的结果，而必须建立在设备授权与用户主权之上。也正因为如此，Mira 才能在保持主动感知能力的同时，不滑向对用户生活的过度侵入。它不是在替用户接管世界，而是在用户允许的范围内，帮助用户更温柔地与世界相处。从更专业的角度看，这一层也体现了设备主权（device sovereignty）与数据最小化（data minimization）的统一。

**核心原则**

Mira 感知你，是为了照顾你，而不是为了监控你。它的设计原则不是“尽可能多地获取信息”，而是“在需要的时候刚刚好地理解，在合适的时候刚刚好地行动”。真正值得信任的陪伴，不是没有边界，而是始终知道边界在哪里。用户始终是自己数据、设备和生活边界的唯一主人。❤

# Mira 发布策略选项对比

## 1. 文档目的

这份文档整理了本次对话里关于 `Mira_Released_Version` 的三种发布策略讨论，用来回答一个核心问题：

- 是先发布一个最小可运行的 Mira 核心包
- 还是尽量原样搬运当前系统
- 还是做成一个分层、可扩展、适合后续独立成仓库的正式发布版

结论先说：

- `1. Core only` 最轻
- `2. Full deployable` 最全
- `3. Layered release` 最适合作为正式开源发布架构

## 2. 方案一：Core only

`Core only` 的目标是发布一个最小但成立的 Mira。

它通常只包含：

- Mira persona workspace
- OpenClaw 配置模板
- 最基础的 skills / plugins 骨架
- 最小部署路径
- 少量示例

### 优点

- 结构最干净
- 最容易让社区理解 Mira 是什么
- 最容易单独拆成新仓库
- 最不容易把私有运行态和历史实验带进发布版

### 缺点

- 生态能力会明显变轻
- 很多真实能力只能退化成接口、示例或说明文档
- 更像“核心框架发布版”，不是“系统总包”

### 适用场景

- 想先把 Mira 作为开源产品概念发布出去
- 想先做一个稳定、清楚、低负担的基础版本

## 3. 方案二：Full deployable

`Full deployable` 的目标是尽量把当前 repo 里已经能工作的系统整体带出去。

它通常会包含：

- Mira persona/workspace
- OpenClaw 配置和桥接
- Home Assistant 能力
- Rokid 路径
- Wearable / Mi Band 路径
- Printer bridge
- Apple 相关支持
- 各类 app、service、脚本和部署说明

### 优点

- 最接近“把当前系统整体搬出去”
- 对 demo、展示、合作说明最直接
- 外部复现完整体验的路径最短

### 缺点

- 目录会明显变重
- 环境依赖和硬件依赖会一起放大
- 维护成本高
- 很容易带入只对当前私有环境成立的内容

### 适用场景

- 目标是系统级移交
- 优先追求“完整还原”，而不是“最优雅的开源结构”

## 4. 方案三：Layered release

`Layered release` 的目标不是简单折中，而是把发布版做成“核心稳定、扩展分层、适合长期演进”的体系。

它的基本思路是：

- `core` 必须能独立成立
- `modules` 是官方扩展层
- `hardware` 是设备接入层
- `apps / services` 只收真正独立可运行的外围组件
- `deploy / readme / docs` 也按层次组织

### 优点

- 最适合单独拆仓
- 最适合社区理解和按需接入
- 体系感最强
- 最容易长期维护和扩展

### 缺点

- 前期整理成本最高
- 需要更明确的边界设计
- 不能只复制文件，必须重写一部分发布文档

### 适用场景

- 目标是正式公开版本
- 未来要持续接新硬件、新插件和新模块
- 希望仓库同时具备可读性、可部署性和可演进性

## 5. `3` 和 `1` 的关系

`3` 可以理解为“以 `1` 为核心内核”，但它不只是 `1` 的简单增量。

更准确的关系是：

- `1` 是“最小可发布核心”
- `3` 是“以最小核心为中心、从第一天就按模块化发布设计好的正式版本”

因此：

- 在内容上，`3` 会包含 `1`
- 在架构思路上，`3` 比 `1` 更强调边界、依赖方向、模块插口和长期治理

## 6. 推荐方案

如果目标是后续把 `Mira_Released_Version` 变成独立仓库，推荐 `3. Layered release`。

原因很直接：

- `1` 太轻，会削弱 Mira 的系统性
- `2` 太重，会把发布版做成一个难维护的大包袱
- `3` 最符合“分类齐全、便于开源社区开放和分享、便于 OpenClaw 部署、能继承软硬件链接和 skill/plugin 体系”的目标

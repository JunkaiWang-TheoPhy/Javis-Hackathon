A release architecture built around a clean core, optional modules, hardware adapters, and deployment/documentation layers.

Layer Model

1. Core

The smallest independently deployable Mira package: workspace, prompts, baseline config, core skills/plugins, and a minimal example.

2. Modules

Optional capabilities grouped by domain: Home Assistant, Rokid, Printer, Apple, Wearable, Local Device Bridges.

3. Hardware

Concrete hardware-facing adapters, desktop/mobile sidecars, device-specific setup, and capability notes.

4. Deploy + Docs

Installation templates, environment examples, scenario walkthroughs, and English release documentation.

Dependency Direction

readme/ + docs/
        │
deploy/ examples/
        │
     modules/
        │
     hardware/
        │
       core/

Rules: core stands alone; modules depend on core; hardware exists to support modules; examples and deploy recipes compose from lower layers; docs explain all layers.

Responsibility Boundaries

Core Boundary

Owns: Mira persona, workspace contract, baseline OpenClaw config, minimal skills/plugins, and the smallest runnable deployment path.

Must not own: vendor-specific integrations, device-side setup, hardware credentials, or ecosystem-specific behavior trees.

Dependency rule: core stands alone and exposes stable interfaces upward; nothing in core should require Home Assistant, Rokid, Apple, or a specific wearable to exist.

Modules Boundary

Owns: domain capabilities built on top of core, such as Home Assistant, Rokid, wearables, printer workflows, and messaging integrations.

Must not own: Mira's base identity, core runtime assumptions, or raw device adapter internals that belong to hardware-facing packages.

Dependency rule: modules may depend on core and coordinate hardware/app/service pieces, but should remain optional and removable without breaking core.

Hardware Boundary

Owns: device adapters, sidecars, local scripts, pairing/setup instructions, capability matrices, and OS/device-specific operational details.

Must not own: product narrative, cross-domain orchestration policy, or long-lived user-facing behavior definitions that should live in core or modules.

Dependency rule: hardware supports upper layers as an implementation surface; it should be swappable and should not become the place where Mira's product logic is defined.

What Goes Where

Inside Core
  • Mira persona workspace
  • OpenClaw config templates
  • Core Lingzhu/Mira path
  • Minimal deployable example
  • Sanitized baseline skills/plugins
Outside Core
  • Home Assistant ecosystem packs
  • Rokid companion apps/services
  • Mi Band / wearable bridges
  • Printer and camera bridges
  • Apple integration assets
